# Crypto Digest: Lessons Learned & Best Practices

> Written at project maturity (February 2026) after the system reached stable production. This document captures what went wrong, what worked, and what future agents should internalise before building similar pipelines.

---

## 1. Understand Your Rate Limits Before Writing a Single Line of Code

The most expensive mistake in this project was designing a concurrent processing architecture (8 threads, 0.5s sleep between calls) against an API plan that allowed 5 requests per minute. The result was a 68% failure rate on the first production run — 42 out of 62 items were silently dropped because retries exhausted themselves against an immovable rate ceiling.

**The lesson:** before you write any code that calls an external API, determine the exact rate limits for your plan tier. Then design your concurrency model around the *actual* constraint, not an assumed one. A simple division — 5 requests per minute means 1 request every 12 seconds — should have been the first constant written. Instead, it was discovered in production through hundreds of 429 errors.

**Best practice:** add a comment at the top of any file that makes API calls stating the rate limit assumption. When the assumption is wrong, you know exactly where to update. Something as simple as `# Rate limit: 5 req/min on claude-sonnet-4-6 (free tier)` would have prevented the initial architecture from ever being concurrent.

## 2. Never Parse LLM Output With Rigid Regex

The batching optimisation reduced API calls from 62 to 7, but introduced a new problem: parsing Claude's multi-item responses back into individual summaries. The initial parser used a narrow regex that only matched `1. text` and a few variations. In practice, Claude returned responses using markdown headers (`## Image 1`), bold labels (`**Image 1:**`), paragraph-based layouts with no numbering, and other formats the regex never anticipated.

The result was that 11 out of 14 image descriptions were successfully generated by the API but thrown away by our own parser. The API calls succeeded; our code failed to read the answers.

**The lesson:** LLM output is inherently variable. Any parser that consumes it must be designed with multiple fallback strategies, not a single regex. The fix was a three-tier approach: (1) broad regex matching many numbered formats, (2) paragraph-based splitting as a fallback when fewer than half the expected items are found, (3) single-item batches bypass parsing entirely and use the raw response. Future systems should plan for this variability from the start.

**Best practice:** when batching multiple items into a single LLM call, include explicit format instructions in the prompt ("Format your response EXACTLY as: 1. [summary] 2. [summary]") and still build a fault-tolerant parser. Trust neither the prompt nor the regex alone.

## 3. Design for the Constraint, Not Around It

The original architecture tried to maximise throughput by processing all items concurrently and relying on the Anthropic SDK's built-in retry logic to handle rate limits. This is designing *around* the constraint rather than *for* it. The SDK retried each failed request 2-3 times, but with 8 threads competing for 5 slots per minute, the retry storms made things worse — each retry consumed rate limit capacity that other threads needed.

The correct approach, implemented in the optimisation, was to accept the rate limit as a design parameter: pre-filter to reduce total items, batch to reduce total calls, and process sequentially with explicit delays. Total API calls went from 62 to 7. No retries needed. No 429 errors.

**Best practice:** when operating under hard constraints (rate limits, quotas, cost budgets), design a pipeline that fits *comfortably* within the constraint rather than one that pushes against it and relies on error handling to recover.

## 4. Pre-Filter Before You Spend Tokens

Of the 62 scraped messages, 50 came from a single image-heavy channel (ahboyashreads). Each image required a separate vision API call in the original design. The pre-filter phase, which uses zero API calls, scored items by text length, URL presence, and channel diversity, then kept only the top 20. This simple heuristic reduced the workload by 68% before any money was spent.

**The lesson:** not all content is equally valuable. A lightweight local scoring step — even one as simple as "does this item have text content?" — can dramatically reduce API costs and improve output quality by focusing the model's attention on higher-signal items.

**Best practice:** any pipeline that processes variable-volume input through a paid API should have a zero-cost filtering stage. Score, rank, cap. The scoring function does not need to be sophisticated; it needs to exist.

## 5. Telegram Markdown Is Not Standard Markdown

The brief was successfully generated but failed to send because Telegram's legacy Markdown parser rejected an unmatched formatting entity. The `send_brief.py` code tracked `**` for bold (standard Markdown) when Telegram uses `*` for bold and `_` for italic. This mismatch meant unclosed entities were never detected.

Additionally, URLs containing underscores can trigger Telegram's italic parser, causing cascading parse failures that are difficult to debug from the error message alone ("can't find end of the entity starting at byte offset 2146").

**Best practice:** when targeting a specific platform's message format, test with that platform's actual parser, not your assumption of what it supports. For Telegram specifically, always sanitise output by ensuring every `*`, `_`, and `` ` `` marker appears an even number of times per message chunk.

## 6. Deduplication Is a Synthesis Problem, Not a Filtering Problem

When the system dropped most of its items due to rate limiting, the few that survived were processed individually and often covered the same underlying news event. The compile step (Claude Opus) then faithfully reported each summary as a separate item, producing a brief that repeated the same story three times with near-identical wording.

The fix was adding explicit deduplication instructions to the compile prompt. But the deeper lesson is that deduplication should be handled at the synthesis layer (the brief compiler), not the ingestion layer. Multiple sources covering the same event is actually valuable — it provides corroboration and additional detail. The problem is only when those multiple angles are presented as separate items rather than merged into one richer entry.

**Best practice:** instruct your synthesis model to merge overlapping items rather than trying to deduplicate upstream. The synthesis model has the context to judge overlap; a simple hash or string comparison does not.

## 7. Log Enough to Debug, Not So Much That You Drown

The initial logs were noisy — hundreds of lines of image download progress and retry attempts. But they were missing the one thing that mattered: what the API actually returned when parsing failed. After adding a response preview log (`logger.warning("Response preview: %.300s", combined)`), the parsing issue would have been diagnosable in seconds rather than requiring a code review.

**Best practice:** log the *shape* of API responses (length, first N characters, structure) in addition to success/failure status. When a downstream parser fails, you need to see what it was trying to parse.
